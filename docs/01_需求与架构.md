# AI课堂助手 — 需求与架构设计

## 1. 项目概述

**项目名称**：AI课堂助手 (AIAnswer)  
**目标**：在上课过程中实时监听老师讲课，自动检测疑似提问的语句，调用AI生成口语化、简短的回答，辅助学生快速组织思路。

## 2. 核心需求

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| R1 | 实时录音 | P0 | 持续从麦克风采集音频，后台运行 |
| R2 | 流式语音转文本 | P0 | 音频流实时送入云端ASR，返回文本流（边说边出字） |
| R3 | 提问检测 | P0 | 检测到疑似提问语句时触发AI回答流程 |
| R4 | AI回答生成 | P0 | 将提问文本作为上下文发送给LLM，附加系统提示词 |
| R5 | 流式展示结果 | P0 | AI回答边生成边显示，降低等待感 |
| R6 | 用户自备API Key | P0 | LLM API Key由用户在设置界面自行填写 |
| R7 | 低功耗适配 | P1 | 适配笔记本离电场景，不跑本地大模型 |
| R8 | 完整记录保存 | P1 | 可选保存整节课的完整文本记录 |
| R9 | 音频设备选择 | P2 | 允许用户选择输入麦克风设备 |

### 提问触发词库（初始版本）
```
问个问题, 请一位同学, 谁来回答, 怎么看, 有什么想法, 
思考一下, 对吗, 是不是, 请回答, 哪位同学, 你的观点,
如何理解, 怎么看待, 有什么见解
```

### LLM系统提示词（固定）
```
回答该提问，口语化并简短精炼
```

## 3. 技术选型

### 3.1 约束条件
- **设备**：笔记本电脑，离电使用
- **算力**：无GPU，不能跑本地大模型
- **网络**：上课环境一定联网
- **延迟**：端到端延迟 < 5秒（可接受）
- **模式**：全流式处理，避免轮询

### 3.2 技术栈

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 语言 | Python 3.10+ | 异步生态成熟，开发快 |
| UI | tkinter | 轻量、不耗电、系统自带、足够用 |
| 音频采集 | pyaudio | 稳定成熟，支持流式chunk读取 |
| 异步核心 | asyncio | 单线程并发处理音频、WebSocket、HTTP |
| 流式ASR | 讯飞语音听写 WebSocket API | 中文课堂场景识别率极高，有免费额度，支持流式 |
| LLM调用 | aiohttp + SSE | 标准OpenAI兼容格式，通吃各家API |
| 配置存储 | JSON文件 | 简单够用，无需数据库 |

### 3.3 为什么不选本地ASR

| 方案 | 离电续航 | 准确率 | 流式延迟 | 结论 |
|------|----------|--------|----------|------|
| 本地qwen0.6b/SenseVoice | CPU高负载，风扇响、耗电快 | 一般 | 1-3s | ❌ 上课体验差 |
| 微信/输入法语音 | 无开放API，无法自动集成 | — | — | ❌ 不可行 |
| 云端流式ASR | 本地仅IO，几乎不耗电 | 极高 | 200ms-1s | ✅ 最优解 |

## 4. 系统架构

### 4.1 模块拓扑

```
┌─────────────────────────────────────────────────────────────┐
│                     UI 层 (tkinter)                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │  实时字幕区   │  │  AI回答区     │  │   设置面板        │  │
│  │ (ASR文本滚动) │  │ (LLM流式输出) │  │ (API Key/ASR配置)│  │
│  └──────────────┘  └──────────────┘  └──────────────────┘  │
└────────────────────┬────────────────────────────────────────┘
                     │  asyncio Queue / Event
┌────────────────────▼────────────────────────────────────────┐
│                  核心引擎 (asyncio 事件循环)                   │
│                                                             │
│   ┌─────────────┐      ┌──────────────┐      ┌───────────┐ │
│   │ AudioCapture │─────→│ StreamingASR │─────→│ TextBuffer│ │
│   │ (pyaudio)    │ chunk│ (WebSocket)  │ text │ (滑动窗口) │ │
│   └─────────────┘      └──────────────┘      └─────┬─────┘ │
│                                                    │       │
│                                             ┌──────▼──────┐│
│                                             │TriggerDetector│
│                                             │(关键词/正则) ││
│                                             └──────┬──────┘│
│                                                    │命中提问│
│                                             ┌──────▼──────┐│
│                                             │ LLMClient   ││
│                                             │ (SSE流式)   ││
│                                             └─────────────┘│
└─────────────────────────────────────────────────────────────┘
```

### 4.2 全流式数据流

1. **AudioCapture** 以 200ms 为 chunk 持续读取麦克风
2. **StreamingASR** 通过 WebSocket 实时推送音频 chunk
3. **ASR服务端** 实时返回文本片段（如"下面"→"下面我来"→"下面我来问个问题"）
4. **TextBuffer** 维护最近 60 秒文本的滑动窗口
5. **TriggerDetector** 扫描缓冲区，命中触发词后提取该句及前一句作为上下文
6. **LLMClient** 将上下文文本以 SSE 流式发送给 LLM API
7. **UI** 左侧实时追加字幕，右侧流式显示AI回答

### 4.3 延迟预算

| 环节 | 预算延迟 |
|------|----------|
| 音频采集缓冲 | 200ms |
| 网络传输 + ASR识别 | 300-800ms |
| 提问检测（本地正则） | < 10ms |
| LLM首字延迟 | 500ms - 2s |
| **总延迟（到首字输出）** | **~1.5s - 3.5s** |

符合 < 5s 要求。

## 5. 非功能性需求

- **稳定性**：单模块异常不应导致整个程序崩溃，需有降级策略
- **隐私**：ASR音频流不本地存储，LLM上下文仅在内存中暂存
- **可配置**：触发词库、API参数、音频设备可配置
- **日志**：关键节点打印日志，便于上课后排查问题
